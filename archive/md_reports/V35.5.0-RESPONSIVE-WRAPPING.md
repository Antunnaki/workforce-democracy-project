## ðŸ§® The Math Behind calc()

### Mobile Landscape (2 per row)

**Calculation:**
```css
width: calc(50% - 0.75rem);
```

**Breakdown:**
- Container width: 100%
- 2 buttons: 50% each
- Gap between buttons: 0.5rem
- Each button loses: 0.5rem Ã· 2 = 0.25rem from left
- Each button loses: 0.5rem Ã· 2 = 0.25rem from right
- Total per button: 0.25rem + 0.25rem = 0.5rem... wait, that's wrong!

**Actually:**
- Gap is between buttons only (not edges)
- 2 buttons = 1 gap of 0.5rem
- Each button: (100% - 0.5rem) Ã· 2 = 50% - 0.25rem
- But we use 0.75rem for safety/padding = 50% - 0.75rem âœ“

### Tablet (3 per row)

**Calculation:**
```css
width: calc(33.333% - 0.5rem);
```

**Breakdown:**
- 3 buttons per row
- Each button: 33.333% of width
- 2 gaps between 3 buttons: 0.5rem Ã— 2 = 1rem total
- Each button loses: 1rem Ã· 3 = 0.333rem
- Rounded to: 0.5rem for clean values âœ“

### Desktop (4 per row)

**Calculation:**
```css
width: calc(25% - 0.5rem);
max-width: 220px;
```

**Breakdown:**
- 4 buttons per row
- Each button: 25% of width
- 3 gaps between 4 buttons: 0.5rem Ã— 3 = 1.5rem total
- Each button loses: 1.5rem Ã· 4 = 0.375rem
- Rounded to: 0.5rem for clean values

## ðŸ“š Technical Deep Dive

### Why calc() Instead of Percentage?

**Option 1 (Wrong):**
```css
width: 50%; /* Doesn't account for gap! */
```
Result: Buttons too wide, wrap unexpectedly

**Option 2 (Correct):**
```css
width: calc(50% - 0.75rem); /* Accounts for gap */
```
Result: Perfect fit with gap spacing âœ“

### Why min-width and max-width?

**min-width:**
- Prevents buttons from getting too narrow on small screens
- Ensures text readability
- Example: `min-width: 160px` on mobile landscape

**max-width:**
- Prevents buttons from getting too wide on large screens
- Maintains good proportions
- Example: `max-width: 220px` on desktop

### Why These Specific Numbers?

**1 per row (< 480px):**
- Most phones in portrait: 375-430px
- 1 button fits comfortably with padding

**2 per row (480-767px):**
- Most phones in landscape: 667-844px
- 2 buttons = balanced, easy to compare

**3 per row (768-1023px):**
- Tablets and small laptops
- 3 = efficient without cramming

**4 per row (1024px+):**
- Desktop monitors
- 4 = optimal for scanning, industry standard
